### 2.1 应用图标系统

**问题**：
```
Electron 用默认图标 → 安装后桌面/任务栏显示 Electron logo
托盘图标是硬编码 16×16 粉色 data URL → 模糊
没有 favicon → 浏览器标签页空白
```

**方案**：
```
SVG 源文件（build/icon.svg）→ sharp 生成 4 种 PNG
  → build/icon.png (1024×1024, electron-builder 源)
  → build/icon-256.png (256×256)
  → public/favicon.png (64×64, 浏览器标签页)
  → electron/tray-icon.png (32×32, 系统托盘)
electron-builder 从 icon.png 自动生成 .ico (Windows) + .icns (macOS)
gen-tray-icon.cjs 升级为优先用 sharp，保留手工 PNG 回退
```

**改了什么**：

| 文件 | 变更 |
|------|------|
| `build/icon.svg` | 🆕 1024×1024 SVG — B站粉圆角方块 + 白色下载箭头 |
| `build/icon.png` | 🆕 sharp 生成的 1024×1024 PNG（52KB） |
| `build/icon-256.png` | 🆕 256×256 备用 |
| `public/favicon.png` | 🆕 64×64 浏览器标签图标 |
| `electron/tray-icon.png` | 🆕 32×32 托盘图标（覆盖旧的像素画版） |
| `scripts/gen-icons.cjs` | 🆕 sharp 批量转换脚本：SVG → 4 种 PNG |
| `electron/gen-tray-icon.cjs` | 🔧 优先用 sharp 读 SVG，保留手工 PNG 回退 |
| `package.json` | 🔧 +`"icon": "build/icon.png"` 到 build 字段；+`"gen-icons"` 脚本；`build`/`build:electron` 前置 `npm run gen-icons` |

**设计**：
- 主色：B站品牌粉 `#FB7299`，圆角方块底板
- 前景：白色下载箭头（简约线条）
- 缩放到 16×16 时仍可辨识

### 2.5 Cookie 自动刷新

**问题**：
```
Cookie 过期 → StatusBar 显示"会话已过期" → 只能手动重新登录
B站提供 refresh_token 接口但未使用
```

**方案**：
```
启动时 validateCookie → 检测到 -101（已过期）
  → 尝试 refreshCookie()
    → POST /x/passport-login/web/cookie/refresh (csrf + refresh_csrf)
    → 成功 → 提取新 cookie → persist + setGlobalCookie
    → 失败 → 回退到显示"会话已过期"（原有行为）
网络容错: 网络错误时不过度反应（保持 valid 状态）
```

**改了什么**：

| 文件 | 变更 |
|------|------|
| `src/services/cookie-manager.ts` | +`refreshCookie()` — 调 refresh API → extractRefreshedCookies() → validateLoginStatus → persistCookie |
| `src/services/bilibili-api.ts` | re-export `refreshCookie`；`validateCookie` -101 分支加注释提示 |
| `src/App.tsx` | 启动 cookie init 从 `.then()` 链改为 `async/await`；过期时先 `refreshCookie()`，成功则 `setGlobalCookie` + `setLoginInfo`，失败则 `setCookieStatus('expired')` |

**容错哲学（继承自 §4.7）**：
```
网络错误 ≠ Cookie 过期
fetch 失败/超时 → 保留 valid 状态
仅 API 明确返回 -101 时才尝试刷新
```

### 2.6 FFmpeg 自动下载引导

**问题**：
```
FfmpegBanner 只有一个"下载 FFmpeg"外链 → ffmpeg.org
用户需自行选版本、下载、解压、设 PATH
全程手动，新手门槛高
```

**方案**：
```
检测 OS + arch → 从 gyan.dev CDN 下载 FFmpeg essentials zip
  → PowerShell Expand-Archive 解压
  → 递归查找 ffmpeg.exe → 复制到 ffmpeg/ 目录
  → 重新检测 → 横幅消失
全流程 Node.js 内置模块（https + fs + child_process），零额外依赖
下载中显示进度条 + 百分比
失败时保留手动下载外链作为回退
```

**改了什么**：

| 文件 | 变更 |
|------|------|
| `electron/ffmpeg-downloader.cjs` | 🆕 核心下载器 — 平台检测 → https 下载（跟随重定向、10 分钟超时）→ PowerShell unzip → 查找 exe → 复制到 `ffmpeg/`→ 清理临时文件 |
| `electron/main.cjs` | +`ipcMain.handle('ffmpeg:download', ...)` — 调下载器，桥接进度事件到渲染进程 |
| `electron/preload.cjs` | +`downloadFfmpeg()` + `onFfmpegDownloadProgress()` |
| `src/electron.d.ts` | +`FfmpegDownloadProgress` / `FfmpegDownloadResult` 接口 |
| `src/stores/ffmpegStore.ts` | +`downloadPhase` / `downloadProgress` / `downloadMessage` / `downloadError`；+`downloadFfmpeg()` action — 注册进度监听 → 调 IPC → 成功自动 re-check → 失败 set error |
| `src/components/FfmpegBanner.tsx` | 🆕 三种 UI 状态：idle（"自动安装 FFmpeg"按钮）/ downloading（Spinner + 进度条 + 百分比 + 可关闭）/ error（错误信息 + 重试 + 手动下载回退） |

**下载源**：
```
Windows x64: https://www.gyan.dev/ffmpeg/builds/ffmpeg-release-essentials.zip
其他平台: 提示前往 ffmpeg.org 手动下载
```

**进度 IPC 协议**：
```
main.cjs → webContents.send('ffmpeg:download-progress', { phase, progress, message })
preload.cjs → onFfmpegDownloadProgress(callback) → cleanup 函数
ffmpegStore → 注册 callback → 更新 progress 状态 → 渲染到 FfmpegBanner
```

### 2.7 极验验证码接入

**问题**：
```
密码登录返回 data.status === 2（需要验证码）时
  代码直接返回 error: "需要验证码，请使用二维码登录"
用户无法完成密码登录
```

**方案**：
```
登录返回 captchaRequired → 动态加载 GeeTest JS SDK (gt.0.4.9.js)
  → GET /x/passport-login/captcha 获取 gt/challenge
  → initGeetest 渲染滑块弹窗
  → 用户完成验证 → 获取 geetest_challenge/validate/seccode
  → 用预加密的密码 + 验证码 token 重试登录
GeeTest 加载失败 → fallback "请使用二维码登录"
```

**改了什么**：

| 文件 | 变更 |
|------|------|
| `src/services/login-service.ts` | `LoginResult` +`captchaRequired?` +`loginParams?`；`loginWithPassword` 支持可选 `captcha` 参数（二次调用时跳过 RSA 加密）；status=2 不报错，返回 captchaRequired=true + 保存加密参数；+`getCaptchaData()` — 获取极验初始化参数 |
| `src/components/LoginPanel.tsx` | `PasswordLoginTab` 重写验证码流程：`captchaVisible`/`captchaLoading` state → `loadGeetestScript()` 动态注入 CDN script → `startCaptcha()` 初始化 widget → `handleCaptchaRetry()` 用 token 重试 → onError/onClose 清理 + 回退提示 |
| `src/vite-env.d.ts` | +`initGeetest` 到 Window 接口 |

**GeeTest 加载时序**：
```
loginWithPassword → captchaRequired=true + loginParams
  → loadGeetestScript() (动态 <script> 注入)
    → 轮询 window.initGeetest 直到就绪
  → getCaptchaData() 并行获取 gt/challenge
  → window.initGeetest({ gt, challenge, product: 'popup' }, callback)
    → captchaObj.onSuccess → handleCaptchaRetry(challenge, validate, seccode)
    → captchaObj.onError → fallback
    → captchaObj.onClose → 取消
```

**错误处理链**：
```
GeeTest SDK 加载超时（10s） → fallback
getCaptchaData 失败 → fallback
登录重试失败 → 显示错误消息
所有路径最终可回退到 QR 码登录
```

### 2.9 ml 多 P 支持 + 分页泛型化

**问题 A — 多 P**：
```
resolveCidForBvid 只返回 pages[0].cid
收藏夹中多 P 视频下载的是第一 P，其他 P 丢失
```

**问题 B — 分页**：
```
parseStore 中的 mlId/mlPage/mlTotalCount 是 ml 专用命名
ss/md（番剧/课程）无法复用分页逻辑
```

**方案 A — 多 P**：
```
resolvePagesForBvid → 返回全部 VideoPage[] (cid/page/part/duration)
resolveCidForBvid → 薄封装，返回 pages[0].cid（向后兼容）

handleAddToQueue:
  单 P → 直接加入（不变）
  多 P → 全部 P 作为独立下载项加入，标题 "标题 — P2 花絮"

handleBatchAddToQueue:
  单 P → 直接加入
  多 P → 跳过 + Toast "xxx 包含多个分P，请单独添加"
```

**方案 B — 分页**：
```
mlId → paginationId
mlPage → paginationPage
mlTotalCount → paginationTotalCount
setMlPagination → setPagination
所有引用同步更新（parseStore + App.tsx + EpisodeList.tsx + bilibili-api.ts + 测试）
```

**改了什么**：

| 文件 | 变更 |
|------|------|
| `src/services/bilibili-api.ts` | +`VideoPage` 接口；+`resolvePagesForBvid()` — 返回全部 pages；`resolveCidForBvid` 改为薄封装；parseMediaList/loadMoreMediaList 中 store 引用更新为新名称 |
| `src/App.tsx` | `handleAddToQueue` 用 `resolvePagesForBvid`：单 P 直接加、多 P 全部加；`handleBatchAddToQueue` 同样升级：多 P 跳过+Toast；pagination 变量名更新 |
| `src/stores/parseStore.ts` | mlId/mlPage/mlTotalCount/setMlPagination → paginationId/paginationPage/paginationTotalCount/setPagination；reset() 同步 |
| `src/__tests__/stores/parseStore.test.ts` | 6 处字段引用 + 2 处 describe/it 名称更新 |

**多 P 下载项格式**：
```
单 P:  { id: "BV1xx", title: "视频标题" }
多 P:  { id: "BV1xx_p2", title: "视频标题 — P2 番外花絮" }
       { id: "BV1xx_p3", title: "视频标题 — P3 幕后制作" }
```

## 四、遇到的问题与解决

### 4.1 sharp SVG → PNG 生成

- **问题**: 项目没有 public/ 和 build/ 目录，没有 favicon，没有图片处理库
- **解决**: `npm install sharp`；创建 `scripts/gen-icons.cjs` 批量生成 4 种尺寸；`build/` 和 `public/` 目录按需创建；Vite 默认服务 `public/` 目录无需额外配置

### 4.2 托盘图标回退兼容

- **问题**: gen-tray-icon.cjs 升级后依赖 sharp，若 sharp 未安装则托盘图标生成失败
- **解决**: 保留原有手工 PNG 生成逻辑作为 fallback（像素级 B 字母图标）；sharp 可用时自动使用 SVG → 高质量 32×32 PNG

### 4.3 并行代理间的文件隔离

- **问题**: 9 个子代理同时运行，若多个代理改同一文件会产生冲突
- **解决**: 事前分析文件依赖关系，按"互不冲突"分组：
  - 第一轮 5 项：各自改完全不同的文件集合
  - 第二轮 4 项：FFmpeg/GeeTest/Git 互不冲突；多P+分页合并给单一代理（因两者改同一批文件）

### 4.4 ffmpeg-downloader 平台兼容

- **问题**: 需在 Windows 上下载和解压 FFmpeg，不能用额外 npm 包
- **解决**: 纯 Node.js 内置模块（https + fs + child_process）；Windows 用 PowerShell `Expand-Archive` 解压 zip；其他平台提示手动下载；10 分钟超时 + 自动跟随重定向 + 递归查找 ffmpeg.exe

### 4.5 GeeTest SDK 动态加载

- **问题**: GeeTest 是外部 JS SDK，需要动态注入 `<script>` 标签，且加载时机不确定
- **解决**: `loadGeetestScript()` 创建 script 元素 → append 到 document.head → 轮询 `window.initGeetest` 直到就绪（最多 10 秒超时）；加载失败 → fallback 到 QR 码登录提示

### 4.6 测试更新

- **问题**: 分页泛型重命名后，`parseStore.test.ts` 中的 `mlId`/`mlPage` 等引用过时
- **解决**: 6 处字段断言 + 2 处 describe/it 名称同步更新；最终 55 passed / 3 skipped

### 4.7 下载完成后的验证序列

- **问题**: FFmpeg 下载完成后需要触发重新检测，但 IPC 是异步的
- **解决**: `ffmpegStore.downloadFfmpeg()` 内部在下载成功后自动调用 `get().checkFfmpeg()`，无需手动触发；FfmpegBanner 通过订阅 `status` 自动更新 UI

### 4.8 极验重试时避免二次 RSA 加密

- **问题**: 完成极验验证后重试登录时，如果重新调用 `loginWithPassword` 会再次 `getLoginKey` + RSA 加密，浪费一次网络请求
- **解决**: `loginWithPassword` 支持可选的 `encryptedPwd` 参数 + `captcha` tokens；重试时传入第一次加密好的密码，跳过密钥获取和加密步骤
