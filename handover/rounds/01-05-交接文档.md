## 一、项目总览

### 1.1 原始项目与重写动机

**原始项目 BilibiliDown** 是一个用 **Java Swing** 编写的 B 站视频下载器，功能完整但 UI 老旧：

- 硬编码绝对布局（`setBounds(x, y, w, h)` 满天飞）
- 无暗色模式（下载工具多在晚上使用）
- 弹窗满天飞（每个错误一个 `JOptionPane`）
- 无组件化（一个文件几千行）
- 不支持现代设计语言（圆角、阴影、动效）

**目标**：用 **Electron + React + TypeScript + Tailwind** 重写整个前端，逐步将 Java 后端逻辑替换为 TypeScript 原生实现。最终产品应是一个独立的跨平台桌面应用（Windows/macOS/Linux），具备完整的 B 站视频下载能力。

## 二、技术决策

### 2.1 技术栈对比

每个技术选择都有明确的理由和被排除的替代方案。

| 层面 | 选用 | 为什么 | 没选什么 |
|------|------|--------|----------|
| **UI 框架** | React 18 + TypeScript | 组件化、生态最大、类型安全 | Vue 3（偏好）、Svelte（太新生态小） |
| **构建工具** | Vite 5 | HMR 即时生效（10x Webpack）、原生 ESM | Webpack（太慢）、Turbopack（不稳定） |
| **样式方案** | Tailwind CSS 3 + CSS Variables | 工具类开发快、CSS 变量做主题切换零成本 | CSS Modules（无设计系统）、styled-components（运行时开销） |
| **桌面壳** | Electron 43 | 跨平台、Node.js 文件系统访问 | Tauri（Rust 门槛）、NW.js（社区小） |
| **状态管理** | Zustand 5 | 比 Redux 轻 10x、API 简洁、自带 persist 中间件 | Redux（太重型）、Jotai（原子化不适合此场景） |
| **路由** | **无路由** — 纯 Zustand 状态切换 | 桌面应用不需要 URL 路由 | React Router（不必要） |
| **图标** | Phosphor Icons | 1,488 个图标、6 种变体、MIT 许可 | Heroicons（图标少）、Lucide（风格不同） |
| **字体** | Segoe UI Variable（系统原生） | ClearType 渲染最优、不依赖外部加载 | Inter（需下载）、系统默认 |
| **API 架构** | **TypeScript 直调 B 站 API（Vite 代理）** | 不依赖 Java 后端、前端独立工作 | 保留 Java 后端做中转 |
| **下载模式** | **fetch + ReadableStream（浏览器）/ Node.js http + fs（Electron）** | 流式下载、进度精确到字节、支持暂停取消 | WebSocket 流、Java 后端下载 |
| **主题** | **先暗色后亮色** | 下载场景多在晚上 | — |
| **配色** | **Slate 冷灰 + 系统强调色** | 匹配 Windows 11 Mica 材质 | Material Design 色板 |

### 2.2 架构设计原则

1. **数据单向流动**: Store → Component ← 用户操作 → Store 更新 → 重渲染
2. **Service 层无状态**: API 调用和业务逻辑不持有 React 状态，通过 Store 桥接
3. **组件渲染与副作用分离**: 下载引擎（`download-manager.ts`）在 App 生命周期中独立运行，不阻塞 UI
4. **Electron 主进程负责 I/O**: 文件写入、FFmpeg 执行、HTTP 下载均在 main process，renderer 只发 IPC 指令
5. **浏览器模式兜底**: 所有功能在 Electron 和纯浏览器环境下都能工作（浏览器模式降级到 Blob + `<a>.click()`）

### 3.3 数据模型

```typescript
// ── 解析结果（parseStore）──
interface ParsedVideo {

  upName: string                // UP 主 / 番剧名
  views: string                 // "123.4万"（已格式化）
  duration: string              // "3:45"（已格式化）
  date: string                  // "2024-01-15"

  episodeTitle?: string         // 单集标题

// ── 清晰度选项（QualityChip 组件）──
interface QualityOption {
  label: string                 // "1080P60" / "4K" / "仅音频"
  size: string                  // "约 210MB"（估算的文件大小）
  available: boolean            // 该清晰度是否可用
  selected?: boolean            // 是否默认选中
}

// ── 下载队列项（downloadStore）──
interface DownloadItemData {
  id: string                    // "dl_1712345678_a1b2c3"
  title: string
  quality: string               // "1080P60"
  format: string                // "mp4" / "flac"
  totalSize: string             // "210MB"（可动态更新）
  downloadedSize: string        // "163.8MB"
  progress: number              // 0-100
  speed: string                 // "12.3 MB/s"
  eta: string                   // "剩余 8 秒"
  status: 'queued' | 'downloading' | 'paused' | 'completed' | 'failed'
  errorMessage?: string         // 失败原因
  speedHistory?: number[]       // 最近 20 个速度点（Sparkline 用）
  bvid?: string                 // BV 号（解析下载链接必需）
  cid?: number                  // 分 P cid（解析下载链接必需）
}

// ── Cookie 信息（cookie-manager）──
interface CookieInfo {
  raw: string                   // 原始 Cookie 字符串
  parsed: Record<string, string> // key-value map
  isValid: boolean              // 是否处于登录态
  validatedAt: number           // 最后验证时间戳
}

// ── 用户信息（登录接口返回）──
interface BiliUserInfo {
  uid: number
  name: string
  face: string                  // 头像 URL
  level: number
  isVip: boolean
}

// ── 下载链接解析结果（bilibili-api）──
interface DownloadUrlResult {
  quality: number               // 实际获取到的 QN 码
  videoUrl: string              // 视频流 URL（DASH）
  audioUrl: string              // 音频流 URL（DASH，FLV 为空）
  format: 'dash' | 'flv'       // 格式类型
}
```

### 4.2 第二轮：B 站 API 接入 + 下载引擎

**目标**: 让应用"真的能工作"——粘贴链接 → 解析 → 下载。

## 五、数据流详解

### 5.1 解析数据流

```
用户粘贴链接
  │
  ▼
InputBar.handleParse()
  │
  ├─ parseStore.setUrl(url)
  ├─ parseStore.setParsing()          ← status → 'parsing'（显示 Loading spinner）
  │
  ▼
parseBilibiliUrl(input)               ← bilibili-api.ts 主入口
  │
  ├─ identifyInput(s)                 ← 正则识别 7 种格式
  │   ├─ b23.tv? → resolveB23Url()    ← HEAD 跟随重定向 → identifyInput(location)
  │   └─ 返回 IdInfo { type, id }
  │
  └─ dispatchParsed(IdInfo)
      │
      ├─ type='bv' → parseBvVideo(bvid)
      │   ├─ GET /x/player/pagelist?bvid=          → pages[] (含 cid)
      │   ├─ GET /x/web-interface/view?bvid=        → detail (标题/UP主/封面)
      │   ├─ for each page:
      │   │   └─ fetchAvailableQNs(bvid, cid)
      │   │       └─ GET /x/player/playurl?...fnval=4048
      │   │           → accept_quality[] → buildQualityOptions()
      │   └─ 返回 ParsedVideo[]
      │
      ├─ type='av' → parseAvVideo(avId)
      │   └─ GET /x/web-interface/view?aid= → bvid → parseBvVideo()
      │
      ├─ type='ep' → parseEpisode(epId)
      │   ├─ GET /pgc/view/web/season?ep_id= → episodes[].bvid
      │   └─ parseBvVideo() + 番剧标题覆盖
      │
      ├─ type='ss' → parseSeason(ssId)
      │   ├─ GET /pgc/view/web/season?season_id=
      │   ├─ >20集: 用默认 QN 列表（省去逐集查 API）
      │   ├─ ≤20集: fetchAvailableQNs() 每集查
      │   └─ 返回 ParsedVideo[]（带 episodeIndex/episodeTitle）
      │
      ├─ type='md' → parseMedia(mdId)
      │   ├─ GET /pgc/review/user?media_id= → season_id
      │   └─ parseSeason(seasonId)
      │
      └─ type='ml' → parseMediaList(mlId)           ← 第四轮新增
          ├─ GET /medialist/gateway/base/detail?...（需登录 Cookie）
          └─ 返回 ParsedVideo[]（cid 未获取，需后续懒加载）
  │
  ├─ 成功: parseStore.setVideos(videos)   → status='success'
  └─ 失败: parseStore.setError(msg)       → status='error'
  │
  ▼
App.tsx（响应 parseStore 变化）
  │
  ├─ status='idle'    → <EmptyParseState />     "粘贴 B 站链接开始解析"
  ├─ status='parsing' → <ParsingState />        Spinner + "正在解析链接..."
  ├─ status='success' → videos ≤ 3 或 无 episodeIndex:
  │                       <VideoCard /> × N
  │                     videos > 3 且 有 episodeIndex:
  │                       <EpisodeList />        ← 批量操作 UI
  └─ status='error'   → InputBar 内部展示错误卡片 + Toast
```

### 5.2 下载数据流

```
用户点击"加入下载队列"（VideoCard 或 EpisodeList 批量）
  │
  ▼
App.handleAddToQueue(video, quality) / handleBatchAddToQueue(episodes, quality)
  │
  ├─ 构造 DownloadItemData { id, title, quality, bvid, cid, status:'queued', ... }
  └─ downloadStore.addItem(item)  × N
  │
  ▼
DownloadPanel 渲染新队列项（状态: queued，shimmer 骨架屏动画）
  │
  ▼
DownloadManager.processQueue()  ← 每秒轮询 + store subscribe 响应
  │
  ├─ 检查: activeCount < maxConcurrent? (userPrefsStore.maxConcurrent, 默认 3)
  ├─ 取队列中第一个 status='queued' 项
  │
  └─ processDownload(item)
       │
       ├─ Step 1: store.updateItem(id, { status:'downloading', speed:'解析链接中...' })
       │
       ├─ Step 2: resolveDownloadUrl(bvid, cid, qn)
       │   ├─ /x/web-interface/view?bvid=         → 判断普通/PGC
       │   ├─ /x/player/playurl?...fnval=4048      → DASH data
       │   │   或 /pgc/player/web/playurl?...
       │   └─ 编码优选: AVC(7) > HEVC(12) > AV1(13)
       │      音频优先: FLAC > 320kAAC > 64kAAC
       │      → 返回 { videoUrl, audioUrl, format }
       │
       ├─ Step 3: 【Electron 模式】
       │   ├─ 下载视频 → IPC saveToDisk → temp/xxx_video.m4s
       │   ├─ 下载音频 → IPC saveToDisk → temp/xxx_audio.m4s
       │   ├─ showSaveDialog → 用户选保存路径
       │   ├─ FFmpeg merge: ffmpeg -i video.m4s -i audio.m4s
       │   │   -c:v copy -c:a aac -b:a 192k -movflags +faststart output.mp4
       │   └─ 完成 → store.updateItem(id, { status:'completed' }) + Toast
       │
       ├─ Step 3': 【浏览器模式】
       │   ├─ fetch + ReadableStream 流式下载
       │   ├─ 每 500ms: progress/speed/eta → store.updateItem()
       │   ├─ speedHistory (最近 20 点) → Sparkline 渲染
       │   ├─ Blob → URL.createObjectURL → <a>.click()
       │   └─ 完成 → store.updateItem(id, { status:'completed' }) + Toast
       │
       └─ 错误处理
           ├─ AbortError → 用户取消/暂停
           ├─ HTTP Error → 失败 + errorMessage
           └─ 网络错误 → 失败 + Toast 通知
```

## 六、遇到的问题和解决方案

以下是开发过程中遇到的 12 个实际问题，按轮次排列。每个问题记录了现象、根本原因、解决方案和涉及的文件。

### 6.1 B 站 API CORS 跨域拦截（第二轮）

- **现象**：浏览器直接 `fetch('https://api.bilibili.com/...')` 被 CORS 拦截
- **原因**：B 站 API 服务器的 `Access-Control-Allow-Origin` 不包含 `localhost:5173`
- **解决**：`vite.config.ts` 配置开发代理 `/api/bilibili` → `https://api.bilibili.com`，`changeOrigin: true` 重写 Host 头
- **涉及文件**：`vite.config.ts`、`bilibili-api.ts`（API_BASE = '/api/bilibili'）

### 6.2 playurl / season API 返回 -404（第二轮）

- **现象**：错误 cid 的 playurl 返回 -404；部分 season_id 返回 -404
- **原因**：① cid 来源错误；② 部分 season_id 已下架
- **解决**：① 从 pagelist API 获取正确 cid；② ep_id 查询比 season_id 更稳定；③ 大列表（>20集）跳过逐个查询
- **涉及文件**：`bilibili-api.ts`

### 6.3 ParsedVideo 缺少 bvid/cid 导致下载链路断裂（第二轮）

- **现象**：DownloadManager 报错"缺少视频标识（bvid/cid），无法下载"
- **原因**：ParsedVideo 只有 `id` 字段，无法传递 bvid/cid 给下载引擎
- **解决**：ParsedVideo 和 DownloadItemData 新增 `bvid?`/`cid?` 字段，handleAddToQueue 传递
- **涉及文件**：`parseStore.ts`、`bilibili-api.ts`、`DownloadItem.tsx`、`App.tsx`

### 6.4 StatusBar 预存 bug — activeCount 未导入（第二轮）

- **现象**：`tsc --noEmit` 报错 `Cannot find name 'activeCount'`
- **原因**：第一轮遗留 — StatusBar 使用了 activeCount 但未从 store 导入
- **解决**：添加 `const activeCount = useDownloadStore((s) => s.activeCount)`
- **涉及文件**：`StatusBar.tsx`

### 6.5 TypeScript Blob 构造函数类型不兼容（第二轮）

- **现象**：`new Blob(chunks)` 类型错误
- **原因**：TS 5.x 的 `ArrayBufferLike` 联合类型与 `BlobPart` 不兼容
- **解决**：类型断言 `new Blob(chunks as BlobPart[])`
- **涉及文件**：`download-manager.ts`

### 6.6 下载管理器 subscribe 内存泄漏（第二轮）

- **现象**：React StrictMode 下 unmount 后 subscribe 仍在运行
- **原因**：`subscribe()` 返回值（清理函数）未被保存
- **解决**：添加 `unsubStore` 变量保存；`stopDownloadManager()` 中调用
- **涉及文件**：`download-manager.ts`

### 6.7 Cookie 域名不匹配（第三轮）

- **现象**：登录成功后 Cookie 未存储，后续请求不带 Cookie
- **原因**：B 站 Set-Cookie 带有 `Domain=.bilibili.com` + `Secure`，浏览器不为 localhost 存储
- **解决**：Vite 代理 `cookieDomainRewrite` 将域重写为 localhost，`proxyRes` 事件移除 Secure
- **涉及文件**：`vite.config.ts`

### 6.8 Web Crypto API 不支持 PKCS1v1.5（第三轮）

- **现象**：密码登录 RSA 加密始终失败
- **原因**：B 站使用 RSA/ECB/PKCS1Padding，Web Crypto API 只支持 RSA-OAEP。且 `RsaOaepParams` 不含 hash 字段触发 TS2353
- **解决**：安装 `jsencrypt`（纯 JS RSA，支持 PKCS1v1.5）
- **涉及文件**：`login-service.ts`、`package.json`

### 6.9 自定义 MD5 实现错误（第四轮）

- **现象**：所有 MD5 结果与标准值不一致
- **原因**：① `v >> 8` 有符号右移 vs `v >>> 8`；② `>>> 32` = `>>> 0`（JS 位移量取低 5 位），64 位长度高位被截断
- **解决**：放弃自研，换 `spark-md5`（3KB）
- **涉及文件**：`md5.ts` 完全重写、`package.json`

### 6.10 QualityChip Props API 不匹配（第四轮）

- **现象**：`<QualityChip options={...} onChange={...} />` 报 TS2322
- **原因**：QualityChip 是单芯片组件（接收 option/selected/focused/onClick/onKeyDown），不是批量组件
- **解决**：EpisodeList 改为手动渲染多个 QualityChip，自行管理 selectedQuality + 键盘导航
- **涉及文件**：`EpisodeList.tsx`

### 6.11 FFmpeg 未找到（第三轮）

- **现象**：合并失败，报错"无法启动 FFmpeg"
- **原因**：用户系统未安装 FFmpeg 或不在 PATH
- **解决**：4 级回退查找：bundled → 同级目录 → 常用路径 → PATH。提供 `ffmpeg:check` IPC
- **涉及文件**：`electron/main.cjs`

### 6.12 ml 收藏夹解析缺少 cid（第四轮）⚠ 未完全解决

- **现象**：收藏夹视频加入队列后下载报错"缺少 cid"
- **原因**：medialist API 返回 bvid 但不含 cid。cid 需单独调 pagelist
- **解决**：标记为已知限制。后续方案：加入队列时懒加载 cid
- **涉及文件**：`bilibili-api.ts`
