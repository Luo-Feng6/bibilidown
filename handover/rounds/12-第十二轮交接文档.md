## 三、改动清单

### 3.1 下载系统 — 核心修复（最复杂，改动最大）

**背景**：用户扫码登录成功后，解析 BV 号正常（标题/封面/UP主都能看到），但点击下载一直失败。

**问题链**（多重原因串联）：

```
用户登录 → Cookie 存 localStorage ──✗──→ 浏览器 cookie jar 是空的
                                              ↓
fetch(/api/bilibili/...) ──→ headersForApi 手动设 Cookie 头
                                     ↓
                          浏览器静默丢弃（Cookie 是禁止头！）
                                     ↓
                          B站 API 收不到 Cookie → 返回低清/空数据
                                     ↓
                          resolveDownloadUrl → "未找到可用链接" → 失败
```

即使 URL 解析成功，浏览器直接 fetch B站 CDN（跨域），CORS 被拒，下载也失败。

**修复措施**（4 个文件联动）：

| 文件 | 修复 | 说明 |
|------|------|------|
| `App.tsx:117-127` | 启动时将 localStorage cookie **注入 document.cookie** | 浏览器 cookie jar 有货，Vite proxy 才能自动转发 |
| `bilibili-api.ts:161-176` | `headersForApi` 浏览器模式不再手动设 `Cookie`/`Origin` | 这两个是浏览器 Fetch API 的**禁止头**，设了也白设 |
| `bilibili-api.ts:201-203` | `apiGet` 添加 `credentials: 'include'` | 确保同源请求携带 cookie |
| `bilibili-api.ts:177-199` | `headersForPgc` 同上处理 | PGC（番剧）API 也用同样逻辑 |

**Electron 生产模式隐患**：`API_BASE = '/api/bilibili'` 在 Electron 加载本地文件时，fetch 到 `file:///api/bilibili/...` 不存在。已记录但未修复（需要协议注册或 IPC 路由）。

## 四、核心问题深度分析

### 4.1 浏览器 Cookie 头被静默丢弃

**问题**：`headersForApi()` 里写 `h['Cookie'] = globalCookie`，然后 `fetch(url, { headers })`。浏览器收到后**静默删除** Cookie 头，请求不带任何认证信息到达 B站 API。

**为什么之前没发现**：Vite proxy 的 `cookieDomainRewrite` 把 B站登录 cookie 的域名从 `.bilibili.com` 改成了 `localhost`，浏览器自动在同源请求中包含这些 cookie。所以如果用户刚登录完就下载，cookie 在浏览器 jar 里，能工作。但关掉浏览器再打开，localStorage 有 cookie 但浏览器 jar 是空的，就彻底失败。

**解决思路**：
1. 认识到 `Cookie` 和 `Origin` 是 Fetch API 的**禁止头**（forbidden headers），浏览器强制不允许 JS 设置
2. 在 `App.tsx` 初始化时，把 localStorage 的 cookie 字符串解析后逐条写入 `document.cookie`
3. `headersForApi` 区分浏览器/Electron 环境，浏览器模式不设 Cookie（反正设了也没用），Electron 模式保留（IPC 走 Node.js HTTP 模块，不受浏览器限制）

**涉及的核心概念**：
- [Fetch Standard: forbidden header names](https://fetch.spec.whatwg.org/#forbidden-header-name)
- Vite proxy 的 `cookieDomainRewrite` + `changeOrigin`
- 浏览器自动 cookie 管理 vs 手动设置

### 4.2 浏览器 CORS 跨域下载

**问题链**：B站 API 返回的视频流 URL 是 CDN 地址（如 `upos-sz-mirrorcos.bilivideo.com`），浏览器 `fetch()` 直连这个跨域 URL → B站 CDN 不返回 `Access-Control-Allow-Origin: localhost:5173` → CORS 错误 → 下载失败。

**解决方案**：双重降级策略
1. 先尝试 `fetch()` — 部分 CDN 节点支持 CORS，能拿到 Blob 和进度
2. 失败后降级为 `<a download>` — 浏览器原生下载机制绕过 CORS，但无进度条

### 4.3 下载失败排查方法论

本轮通过 `Agent` 工具启动专项分析 agent，审查了完整下载链路中的 **10 个潜在问题**，按严重程度排序：
1. **CRITICAL**: Cookie 头被浏览器丢弃
2. **CRITICAL**: `apiGet` 缺 `credentials: 'include'`
3. **HIGH**: Electron 生产模式无 API 后端（`file://` 协议下 `/api/bilibili` 不存在）
4. **HIGH**: `resolveDownloadUrl` 错误信息太模糊（"未找到可用链接" 无法区分过期/权限/地区限制）
5. **HIGH**: Electron 下载不传 Cookie 到 CDN
6. **MEDIUM**: 仅音频时下载流程死锁不报错
7. **MEDIUM**: 浏览器 CDN CORS 问题
8. **MEDIUM**: 双重 cookie 存储系统（`bibilidown-cookies` vs `bibilidown-prefs`）
9. **LOW**: 无重试逻辑
10. **LOW**: 浏览器 DASH 速度计算不准确

其中问题 1、2、7、8 在本轮修复。3、4、5、6 已记录待后续修复。

## 七、核心经验 & 注意事项

### 7.1 浏览器 Fetch API 的禁止头

**最重要的一条经验**：`Cookie`、`Origin`、`Referer` 等是浏览器的**禁止请求头**（forbidden request headers）。`fetch()` 中手动设置这些头会被浏览器**静默删除**，不会报错。

- ✅ 正确做法：让浏览器自动管理 cookie（同源请求自动带，跨域加 `credentials: 'include'`）
- ✅ Vite proxy：通过 `configure.proxyReq` 在服务端设这些头
- ❌ 错误做法：`headers: { Cookie: '...' }` — 白写

### 7.2 localStorage cookie ≠ 浏览器 cookie jar

`document.cookie` 和 `localStorage` 是两个独立的存储。Zustand persist 把 cookie 存到 localStorage，但浏览器 fetch 只看 `document.cookie`。

**必须**在启动时把 localStorage 的 cookie 注入 `document.cookie`：
```ts
for (const pair of cookieStr.split(';')) {
  const [key, ...rest] = pair.trim().split('=')
  document.cookie = `${key}=${rest.join('=')}; path=/; SameSite=Lax`
}
```

### 7.3 浏览器 CDN 下载的 CORS 困境

浏览器 `fetch()` 跨域请求 CDN 文件时，CDN 必须返回 `Access-Control-Allow-Origin`。B站 CDN 通常允许 `bilibili.com` 但不允许 `localhost`。

**两种解决思路**：
1. 降级为 `<a download>` — 丢失进度条但可靠
2. 通过 Vite proxy 代理 CDN 请求 — 但 CDN 域名不固定

本轮选择了方案 1（fetch + 失败降级），简单可靠。

### 7.4 Electron vs 浏览器双模式维护

两个模式有不少差异，未来的开发者需要注意：

| 差异点 | 浏览器 | Electron |
|--------|--------|----------|
| Cookie 传递 | Vite proxy 自动转发 | 需手动设 Cookie 头（IPC → Node HTTP） |
| 文件下载 | `<a download>` 或 fetch+blob | IPC → main 进程 → Node fs |
| 音视频合并 | 不支持（无 FFmpeg） | FFmpeg 自动合并 |
| 文件夹选择 | `showDirectoryPicker` | 原生对话框 |
| API 请求 | 通过 Vite proxy | 直接请求或需自定义协议 |
| 禁止头 | Cookie/Origin 不能手动设 | IPC 到 Node 后可以设 |
