## 二、任务一：ml 收藏夹分页加载

### 2.1 问题描述

> 一个收藏夹如果超过 20 个视频，用户只能看到前 20 个，其余视频无法访问和下载。

**原始行为**（第四轮）：
```
parseMediaList(mlId)
  └─ GET /medialist/gateway/base/detail?media_id={mlId}&pn=1&ps=20
  └─ 如果 totalCount > 20: console.log("收藏夹共 N 个视频，已加载前 20 个")
  └─ 用户无感知，无加载更多入口
```

**目标**: 用户能逐页加载全部视频，UI 上显示明确的"加载更多"按钮。

### 2.2 设计方案

**核心思路**: 首次解析加载首页 20 条 → 存储分页元信息到 parseStore → EpisodeList 底部显示"加载更多"按钮 → 点击触发下一页请求 → `appendVideos` 追加 → 直到全部加载完毕。

**涉及的数据流**:
```
首次解析:
  InputBar → parseBilibiliUrl → dispatchParsed('ml') → parseMediaList(mlId)
    ├─ GET pn=1 → mediasToVideos → 返回 ParsedVideo[0..19]
    └─ 检测 hasMore → setMlPagination(mlId, page=1, total=N)

加载更多:
  EpisodeList "加载更多" 按钮 → App.onLoadMore → loadMoreMediaList()
    ├─ 从 parseStore 读 mlId/mlPage
    ├─ GET pn=2 → mediasToVideos → appendVideos
    ├─ 更新 mlPage/hasMore
    └─ Toast: "已加载 40 / 120 个视频"
```

### 2.3 实现细节

#### 2.3.1 `stores/parseStore.ts` — 新增分页状态

新增 4 个 ml 分页字段 + 3 个 action：

```typescript
// 状态字段
mlId: string | null    // 当前 ml ID，非 ml 类型为 null
mlPage: number         // 已加载到的页码
mlTotalCount: number   // 收藏夹视频总数（来自 API info.media_count）
isLoadingMore: boolean // 是否正在加载更多（按钮 loading 态）

// 新增 actions
setMlPagination(mlId, page, total)  // 初始/更新分页信息
setLoadingMore(boolean)             // 切换 loading 状态
// reset() 已更新：自动清除以上 4 字段
```

**设计考虑**: parseStore 是通用解析 store，加 ml 专用字段略显耦合，但避免了新建独立 store 的过度工程。未来如 ss/md 也需要分页，可提炼为泛型分页状态。

#### 2.3.2 `services/bilibili-api.ts` — 核心改造

**提取复用函数** (`mlHeaders`, `mediasToVideos`):

`parseMediaList` 和 `loadMoreMediaList` 共享 headers 构建和 API 响应 → ParsedVideo 转换逻辑，避免代码重复。

```typescript
// 构建收藏夹请求头（含登录 Cookie）
function mlHeaders(mlId: string): Record<string, string>

// 将 API 返回的 medias[] 转为 ParsedVideo[]
function mediasToVideos(
  medias: any[], mlId: string, collectionTitle: string, startIndex: number
): ParsedVideo[]

const ML_PAGE_SIZE = 20  // 每页条数常量
```

**`parseMediaList` 改造**:
```
原来: pn=1 硬编码 → 只返回 20 条
现在: pn=1 → 检查 hasMore → 设置 parseStore.mlPagination
  ├─ hasMore=true  → setMlPagination(mlId, page=1, total)
  └─ hasMore=false → setMlPagination(null, 0, 0)  // 全部加载完
```

**hasMore 判断逻辑**:
```
优先: API response.data.has_more 字段
Fallback: medias.length >= ML_PAGE_SIZE && totalCount > 已加载数
```

**新增 `loadMoreMediaList()`**（导出给 App.tsx 调用）:
```
1. 从 parseStore.getState() 读 mlId, mlPage, isLoadingMore
2. 防御: mlId 为 null 或已在 loading → return（幂等）
3. 防御: nextPage > maxPage → return（无更多数据）
4. setLoadingMore(true)
5. GET /medialist/gateway/base/detail?...&pn={nextPage}&ps=20
6. 更新分页状态: hasMore? setMlPagination(id, nextPage, total) : setMlPagination(null, 0, 0)
7. appendVideos(newVideos)
8. Toast.info("已加载 40 / 120 个视频")
9. catch: Toast.error → re-throw
10. finally: setLoadingMore(false)
```

**关键技术决策: 动态 import 避免循环依赖**:
`bilibili-api.ts` 静态引用了 parseStore 的类型（`import type { ParsedVideo }`），但在运行时需要调用 `useParseStore.getState()`。为避免循环依赖，使用 `await import('../stores/parseStore')` 和 `await import('../stores/toastStore')` 动态导入。这是 Vite build 警告的来源（信息性，不影响功能）。

#### 2.3.3 `components/EpisodeList.tsx` — 新增"加载更多"按钮

**新增 Props**:
```typescript

```

## 三、任务二：ml cid 懒解析

### 3.1 问题描述

> 主交接文档 §6.12 记录的问题：medialist API 返回 bvid 但不含 cid（分 P 视频流标识），导致收藏夹视频加入下载队列后，下载引擎报错"缺少视频标识（bvid/cid），无法下载"。

**原始行为**:
```
parseMediaList → cid: undefined（硬编码，第 619 行）
↓
用户点击"加入下载队列" → DownloadItemData { cid: undefined }
↓
processDownload → "缺少视频标识（bvid/cid），无法下载" → failed
```

**目标**: 加入下载队列时自动补齐 cid，用户无感知。

### 3.2 设计方案

**核心思路**: 在 `handleAddToQueue` 和 `handleBatchAddToQueue` 中，检测到 `cid == null` 时自动调用 pagelist API 获取 cid，成功后再加入队列。

**分两步**:
1. API 层：提供 `resolveCidForBvid(bvid)` 函数（封装 pagelist API）
2. UI 层：`handleAddToQueue` 和 `handleBatchAddToQueue` 改为 async，加入队列前先 resolve cid

### 3.3 实现细节

#### 3.3.1 `services/bilibili-api.ts` — 新增 `resolveCidForBvid`

```typescript
/**
 * 懒解析 cid：通过 BV 号获取对应视频的默认 cid。
 * 调用 /x/player/pagelist?bvid= 获取分 P 列表，返回第一 P 的 cid。
 */
export async function resolveCidForBvid(bvid: string): Promise<number | undefined> {
  try {
    const url = `/x/player/pagelist?bvid=${bvid}&jsonp=jsonp`
    const pages = await apiGet(url, headersForApi(bvid))
    if (Array.isArray(pages) && pages.length > 0) {
      return pages[0].cid as number
    }
  } catch {
    console.warn(`[resolveCidForBvid] 无法获取 ${bvid} 的 pagelist`)
  }
  return undefined
}
```

**为什么只返回第一 P 的 cid**: 大多数收藏夹视频是单 P。后续如需支持多 P 选择，可扩展返回值包含完整 pages 数组。目前标记为未来改进点。

#### 3.3.2 `App.tsx` — handleAddToQueue async 改造

```typescript
const handleAddToQueue = async (video, quality) => {
  let cid = video.cid
  if (cid == null && video.bvid) {
    cid = await resolveCidForBvid(video.bvid)
    if (cid == null) {
      toastStore.error(`无法获取 ${video.title} 的视频信息，请重试`)
      return  // 不加入队列
    }
  }
  // ... 正常加入队列，cid 已补齐
}
```

#### 3.3.3 `App.tsx` — handleBatchAddToQueue async 改造

```typescript
const handleBatchAddToQueue = async (episodes, quality) => {
  // 1. 收集所有需要解析 cid 的 episode
  const needCid = episodes.filter(ep => ep.cid == null && ep.bvid)

  // 2. 并行解析所有缺失 cid
  if (needCid.length > 0) {
    const results = await Promise.allSettled(
      needCid.map(ep => resolveCidForBvid(ep.bvid!))
    )
    // 写回 ep.cid（直接 mutate ParsedVideo 对象）
    results.forEach((r, i) => {
      if (r.status === 'fulfilled' && r.value != null) {
        needCid[i].cid = r.value
      }
    })
  }

  // 3. 逐个加入队列，cid 仍为 null 的跳过
  let added = 0, skipped = 0
  for (const ep of episodes) {
    if (ep.cid == null) { skipped++; continue }
    addItem({...}) // cid 已补齐
    added++
  }

  toastStore.success(`已添加 ${added} 个视频到下载队列`)
  if (skipped > 0) toastStore.warning(`${skipped} 个视频无法获取下载信息，已跳过`)
}
```

**关键技术决策: `Promise.allSettled` + 直接 mutate**:
- `allSettled`（不是 `all`）：单个视频 cid 解析失败不影响其他
- 直接 `needCid[i].cid = r.value` mutate：ParsedVideo 对象在 parseStore 的数组中，直接修改引用型字段即可生效，不需要重新 setVideos

**用户体验设计**:
- 批量场景下静默并行解析 cid（用户只在最终 Toast 看到结果）
- 失败的视频不阻塞成功的视频
- 清晰的统计数字："已添加 18 个" / "2 个无法获取"

## 四、任务三：HistoryPage 历史记录

### 4.1 问题描述

> 第四轮的 HistoryPage 只有空状态骨架。用户下载过的视频没有记录，想重新下载需要再次粘贴链接。

**原始代码**: `pages/HistoryPage.tsx` 只有一个 `<ClockCounterClockwise>` 图标 + "暂无历史记录" 文案。

**目标**: 
1. 下载完成/失败时自动记录历史
2. 历史页展示记录列表（搜索、筛选、排序）
3. 支持"重新下载"（自动切到下载页并解析）
4. 持久化存储（重启应用后历史不丢失）

### 4.2 设计方案

```
数据流: download-manager 完成下载 → historyStore.addEntry → localStorage persist
UI: Sidebar 切到历史视图 → HistoryPage → useHistoryStore → 渲染列表
重新下载: 点击"重新下载" → App.handleReDownload(bvid) → navigate('download') + parseBilibiliUrl
```

### 4.3 实现细节

#### 4.3.1 `stores/historyStore.ts` — 新建

```typescript
interface HistoryEntry {
  id: string           // downloadItem.id（唯一标识，用于去重）
  title: string        // 视频标题
  bvid?: string        // BV号（重新下载必须）
  quality: string      // 下载清晰度标签
  format: string       // mp4/flac
  size: string         // totalSize
  downloadedAt: number // Date.now() 时间戳
  status: 'completed' | 'failed'
  errorMessage?: string
}
```

**Zustand persist 中间件**:
```typescript
export const useHistoryStore = create<HistoryStore>()(
  persist(
    (set) => ({
      entries: [],
      addEntry: (entry) => set(s => {
        // 去重：相同 id 覆盖旧记录
        const filtered = s.entries.filter(e => e.id !== entry.id)
        // 上限 500 条
        return { entries: [entry, ...filtered].slice(0, 500) }
      }),
      removeEntry: (id) => set(s => ({ entries: s.entries.filter(e => e.id !== id) })),
      clearHistory: () => set({ entries: [] }),
    }),
    { name: 'bilibilidown-history' }  // localStorage key
  )
)
```

**设计决策**:
- `persist` 中间件：无需手动管理 localStorage，Zustand 自动序列化/反序列化
- 500 条上限：防止 localStorage 无限增长
- 去重逻辑：同 id 覆盖（重试下载时更新状态和时间）

#### 4.3.2 `services/download-manager.ts` — 自动记录

新增 `recordToHistory` 辅助函数，在 5 个关节点调用：

| 位置 | 触发时机 |
|------|---------|
| Electron 合并成功 | `recordToHistory(item, 'completed')` |
| Electron FLV/单视频成功 | `recordToHistory(item, 'completed')` |
| Electron/浏览器通用 catch | `recordToHistory(item, 'failed', errorMsg)` |
| 浏览器 DASH 完成（含警告） | `recordToHistory(item, 'completed', '浏览器模式下音视频分离')` |
| 浏览器 FLV/单视频完成 | `recordToHistory(item, 'completed')` |

**关键点**: `recordToHistory` 在 `catch` 块中、AbortError 检查之后调用，确保只记录真正的失败而非用户取消。

#### 4.3.3 `pages/HistoryPage.tsx` — 完整重写

**新增导入**: `PhosphorIcon` 类型（用于 EmptyState 组件类型安全）

**界面结构**:
```
┌─ 标题栏 ──────────────────────── [清空历史] ─┐
│ ClockIcon 历史记录  25 条                     │

│ 🔍 [搜索标题或 BV 号...      ][清除]        │
```

**搜索/筛选实现**:
```typescript
const filtered = useMemo(() => {
  let list = [...entries]
  // 状态筛选
  if (filter === 'completed') list = list.filter(e => e.status === 'completed')
  if (filter === 'failed') list = list.filter(e => e.status === 'failed')
  // 文本搜索（标题 + BV号）
  if (search.trim()) {
    const q = search.trim().toLowerCase()
    list = list.filter(e =>
      e.title.toLowerCase().includes(q) || (e.bvid ?? '').toLowerCase().includes(q)
    )
  }
  // 最新在前
  return list.sort((a, b) => b.downloadedAt - a.downloadedAt)
}, [entries, search, filter])
```

**相对时间格式化**:
```
< 1分钟 → "刚刚"
< 1小时 → "N 分钟前"

```

**空状态两态**:
- 真正无记录: ClockIcon + "暂无历史记录" + 说明文案
- 搜索无结果: MagnifyingGlass + "没有匹配的记录" + "尝试调整搜索或筛选条件"

#### 4.3.4 `App.tsx` — 重新下载接线

## 六、遇到的问题和解决方案

### 6.1 动态 import 导致的 Vite 警告

**现象**: Vite build 输出两条 warning：
```
parseStore.ts is dynamically imported by bilibili-api.ts but also statically imported by App.tsx...
toastStore.ts is dynamically imported by bilibili-api.ts but also statically imported by App.tsx...
```

**原因**: `bilibili-api.ts` 使用 `await import('../stores/parseStore')` 避免循环依赖（parseStore 静态引用了 bilibili-api 的 `QualityOption` 类型），但 App.tsx 又静态 import 了同一个 store。Vite 提示这不会将模块移到独立 chunk。

**解决**: 信息性警告，不影响构建和功能。动态 import 的语义正确——`bilibili-api.ts` 只在运行时需要 store 引用（设置分页状态），不需要在模块初始化时加载。**无需修复**。

### 6.2 EmptyState 组件 Icon 类型不兼容

**现象**: `tsc --noEmit` 报错：
```
TS2322: Type 'ForwardRefExoticComponent<IconProps>' is not assignable to type
'ComponentType<{ size: number; weight: "regular"; style?: CSSProperties }>'
```

**原因**: `EmptyState` 的 `icon` prop 类型声明为 `React.ComponentType<{ size: number; ... }>`，但 Phosphor 图标的 `size` 属性类型为 `string | number | null | undefined`，不兼容 `number`。

**解决**: 改用 Phosphor 官方的 `Icon` 类型：
```typescript
import type { Icon as PhosphorIcon } from '@phosphor-icons/react'

interface EmptyStateProps {
  icon: PhosphorIcon   // ← 直接用 Phosphor 的 Icon 类型
  message: string
  subtitle: string
}
```

### 6.3 分页判断的 hasMore 容错

**问题**: B 站 medialist API 的 `data.has_more` 字段不一定总是存在（不同 API 版本可能省略此字段）。

**解决**: 双重判断逻辑：
```typescript
const hasMore = data?.has_more ?? (
  medias.length >= ML_PAGE_SIZE && totalCount > 已加载数
)
```
优先用 API 返回值，fallback 用数学计算（当页满 + 总数大于已加载）。

### 6.4 批量 cid 解析失败的处理

**问题**: `Promise.all` 失败会丢弃所有结果，`Promise.allSettled` 才能让部分成功。

**解决**: 使用 `Promise.allSettled` + 单独统计 added/skipped：
```typescript
const results = await Promise.allSettled(needCid.map(...))
// 成功的写回，失败的保持 cid=undefined → 后续跳过
results.forEach((r, i) => {
  if (r.status === 'fulfilled' && r.value != null) needCid[i].cid = r.value
})
// 统计: added vs skipped，分别 Toast
```

### 6.5 历史记录大小限制

**问题**: localStorage 容量有限（通常 5-10MB），如果用户下载数千个视频，历史记录可能撑满存储。

**解决**: 500 条硬上限 + Zustand persist 自动序列化。每个 HistoryEntry ~200 bytes，500 条 ≈ 100KB，安全裕度充足。
