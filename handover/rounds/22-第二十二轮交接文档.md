## 四、逐任务详解

### 4.1 README 重写（~70 行 → ~250 行）

#### 为什么要改

旧 README 的问题：
- **功能描述过于简略**：比如「7 种输入」没有列出是哪 7 种，「14 级清晰度」没有列出具体等级
- **没有项目结构**：读者无法一眼看懂代码是怎么组织的
- **技术栈只有一行**：没有版本号、没有链接、没有分层说明
- **没有 FAQ**：常见问题（下载为什么要代理、DASH vs FLV）需要反复解释
- **构建打包命令缺失**：只有开发命令，没有 `build:electron` 流程
- **平台徽章不全**：没有 Electron/React 版本徽章，没有平台支持声明

#### 新增内容的结构设计

我按这个逻辑组织——**先给印象（徽章+对比表）→ 再讲功能（分 6 个类别展开）→ 然后上手（快速开始+项目结构）→ 最后深入（技术栈+FAQ+路线图）**：

```
📛 徽章栏（version / tests / TS / platform / Electron / React / license）
📊 两种使用方式对比表
📥 功能一览（6 个分类，每个都有子项和说明）
  ├── 视频下载（14 级清晰度 / DASH+FLV / 多P / 弹幕 / 字幕 / 下载方案）
  ├── 输入格式（7 种格式 × 示例 × 说明的表格）
  ├── 登录系统（QR / 密码 / 短信 / Cookie 持久化 / 退出）
  ├── 历史记录（自动记录 / 搜索 / 导入导出 / 容量上限）
  ├── 主题与外观（亮暗双主题 / 5色 / CSS变量 / Fluent Design）
  └── 桌面体验（托盘 / 开机自启 / NSIS / 便携版）
🚀 快速开始（环境要求 + npm 命令 + 输出说明）

❓ FAQ（4 个常见问题，用 <details> 折叠）
🗺 路线图
🙏 致谢 + License
```

#### 设计决策

**1. 为什么用 `<details>` 折叠 FAQ 而不是直接展开？**

README 已经很长了。FAQ 放在末尾用折叠式，读者按需展开，不会在中间打断阅读流。

**2. 为什么技术栈用表格而不是一行文字？**

一行文字 `React 18 · TypeScript 5 · ...` 的格式在 3-4 个技术时可以，但本项目有 9 个技术组件。表格可以把每个技术的**名称、版本、官网链接**对齐展示，一眼看完。而且 Markdown 链接让读者可以直接点进去了解不熟悉的技术。

**3. 为什么项目结构要写那么细？**

12 个组件、8 个 service、8 个 store——这是项目的骨架。新接手者最大的困惑就是「文件这么多，从哪看起」。把每个文件的职责用一行中文注释标清楚，相当于给了份小地图。

### 4.2 EpisodeList 分页 bug 修复

#### 背景

INDEX.md 中标注了 🟡 已知 bug：「收藏夹 >3 集翻页不完整」。从 R5 引入 ml 收藏夹分页功能后，这个 bug 就一直存在，只是之前没有专门排查。

#### 排查过程

我先让 explore agent 扫了整个分页链路（EpisodeList.tsx → App.tsx → parseStore.ts → bilibili-api.ts），拿到一份初步的问题清单。然后我自己逐行走了 `parseBilibiliUrl` → `dispatchParsed` → `parseMediaList` → `loadMoreMediaList` 的完整调用链，确认了**两个独立根因**。

整个排查的思路是：分页涉及 **状态写入**（parseMediaList 写入 paginationId/page/totalCount）、**状态读取**（App.tsx 计算 hasMore）、**状态追加**（loadMoreMediaList 读取 paginationId/page 请求下一页）三个环节。逐一排除 → 发现写入和读取各有一个 bug。

#### Bug 1（主因）：跨解析类型的分页状态残留

**调用链**：

```
InputBar.handleParse()
  → parseBilibiliUrl(input)
    → dispatchParsed(idInfo)
      ├── case 'ml': parseMediaList(mlId)    ← 设置 paginationId, page, totalCount
      ├── case 'ss': parseSeason(ssId)        ← ❌ 没有清理分页状态
      ├── case 'bv': parseBvVideo(bvId)       ← ❌ 没有清理分页状态
      ├── case 'ep': parseEpisode(epId)       ← ❌ 没有清理分页状态
      ├── case 'md': parseMedia(mdId)         ← ❌ 没有清理分页状态
      └── case 'av': parseAvVideo(avId)       ← ❌ 没有清理分页状态
    → InputBar 拿到 videos
    → setVideos(videos)                       ← ❌ setVideos 不碰 pagination 字段
```

只有 `parseMediaList` 操作分页状态（`setPagination(mlId, page, totalCount)`），其他 5 个解析函数完全不碰。`setVideos` 也只写 `videos/status/error`，不重置 pagination。

**复现步骤**：

1. 解析一个 >20 集的收藏夹（如 `ml900`，60 集）
   - `paginationId = "900"`，`paginationTotalCount = 60`，`videos.length = 20`
   - `paginationHasMore = "900" != null && 20 < 60` → `true` ✅ 正确
2. 再解析一个 12 集的 ss 番剧
   - `parseSeason` 不碰 pagination → `paginationId` 仍然是 `"900"`
   - `setVideos` 不碰 pagination → `paginationTotalCount` 仍然是 `60`
   - `paginationHasMore = "900" != null && 12 < 60` → `true` ❌ **幽灵按钮！**
3. 用户看到番剧下方出现「加载更多」按钮
4. 点击后 `loadMoreMediaList` 用残留的 `paginationId="900"` 去请求 `/medialist/gateway/base/detail?media_id=900&pn=2` → 把旧收藏夹的视频追加到番剧列表里

**修复**：在 `parseBilibiliUrl` 入口统一清理分页状态。

```typescript
// bilibili-api.ts — parseBilibiliUrl 函数开头新增 3 行
export async function parseBilibiliUrl(input: string): Promise<ParsedVideo[]> {
  // 清理上一次解析的分页状态，避免残留到不同类型（如 ml→ss 出现幽灵"加载更多"按钮）
  const { useParseStore } = await import('../stores/parseStore')
  useParseStore.getState().setPagination(null, 0, 0)

  const s = input.trim()
  // ...
}
```

**为什么放在入口而不是每个 parse 函数里？**

6 个解析类型中，只有 `parseMediaList` 需要分页。在入口统一清理 + `parseMediaList` 按需重设，比在 5 个函数里各加一行清理更可靠（不会漏），也更符合「谁创建谁负责、入口兜底」的模式。

#### Bug 2（次要）：`has_more` 类型陷阱

**位置**：`parseMediaList` 和 `loadMoreMediaList` 各一处。

```typescript
// ❌ 修复前
const hasMore = data?.has_more ?? fallback

// ✅ 修复后
const hasMore = (data?.has_more) || fallback
```

**根因**：`??`（Nullish Coalescing）只在左侧为 `null`/`undefined` 时回退。如果 B 站 API 返回 `has_more: 0`（整数 false），则 `0 ?? fallback` 求值为 `0`，而 `if (0)` 是 falsy → 分页静默失败，不显示「加载更多」按钮。

**对比**：

| API 返回 | `??` 结果 | `\|\|` 结果 | 正确行为 |
|---------|----------|-----------|---------|
| `has_more: true` | `true` | `true` | 显示按钮 |
| `has_more: false` | `false` | fallback | 用 fallback 判断 |
| `has_more: 1` | `1`（truthy） | `1`（truthy） | 显示按钮 |
| `has_more: 0` | `0`（❌ falsy→不显示） | fallback（✅） | 用 fallback 判断 |
| `has_more: undefined` | fallback | fallback | 用 fallback 判断 |
| 无 has_more 字段 | fallback | fallback | 用 fallback 判断 |

`||` 能覆盖 `false`/`0`/`null`/`undefined` 四种异常情况，比 `??` 更健壮。

#### 这个 bug 为什么之前没被发现

- 正常情况下 B 站 API 返回的是布尔值 `true`/`false`，`??` 和 `||` 行为一致
- Bug 1 需要「先解析 ml、再解析其他类型」的特定操作顺序才会触发——大多数测试场景是单次解析
- Bug 2 只在 API 返回非标准类型时才暴露——属于防御性编程范畴

### 问题 2：探索阶段走了弯路

一开始我直接从 EpisodeList 组件和分页 UI 层面去猜 bug（以为是什么条件判断写反了、边界情况没处理），读了几百行代码没发现问题在哪。后来意识到应该追踪**完整数据流**——从 parseStore 的状态写入、到 App.tsx 的派生计算、到 bilibili-api 的 API 请求——把三个环节逐个排查，才定位到分页状态跨解析类型残留的根因。

**经验**：遇到"某个功能看起来能跑但特定场景下坏了"这类 bug，不要盯着 UI 层猜，先画出完整的数据流向图，再逐一检视每个环节的输入输出是否符合预期。
