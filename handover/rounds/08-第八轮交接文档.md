## 二、做了什么

### 已修复的 Bug（共 12 项）

#### Bug 1 🔴 白屏 / 无限重渲染循环
- **表现**：浏览器打开 `localhost:5173` 白屏，Console 报 `Maximum update depth exceeded`
- **根因**：`App.tsx` 使用内联对象 selector `useDownloadStore((s) => ({...}))`，每次渲染创建新对象引用。zustand v5 内部 `useSyncExternalStore + Object.is` 判定"变化"→ setState → 重渲染 → 死循环
- **修复**：拆成独立字段 selector（每个返回稳定函数引用/原语值）；`LoginPanel`/`SettingsPage`/`HistoryPage` 用 `useShallow` 替代无选择器订阅
- **涉及**：`App.tsx`、`LoginPanel.tsx`、`SettingsPage.tsx`、`HistoryPage.tsx`

#### Bug 2 🔴 视频封面：纯黑 / 无加载提示 / 无失败兜底
- **表现**：解析成功后封面区域全黑，无加载提示，图片失效时无占位图
- **修复**：引入三态管理（loading/success/error），`<img>` 始终可见 + loading/error 覆盖层叠加；添加 `referrerPolicy="no-referrer"` 防盗链
- **涉及**：`VideoCard.tsx`

#### Bug 3 🔴 二维码一直显示"已过期"
- **表现**：打开登录面板 → 二维码立即显示"已过期"
- **根因**：① API 域名错误（登录 API 在 `passport.bilibili.com`，但 Vite 代理只配了 `api.bilibili.com`）；② `source` 参数过时：`main-web` → `main-fe`
- **修复**：`vite.config.ts` 新增 `/api/passport` 代理；`login-service.ts` 所有 passport 接口改用正确 base + source
- **涉及**：`vite.config.ts`、`login-service.ts`、`cookie-manager.ts`

#### Bug 4 🟡 扫码成功但无反应（轮询静默失败）
- **表现**：二维码能显示，手机扫码后页面无任何反应
- **根因**：轮询 catch 块是空的 `catch {}`，API 返回格式变化或网络错误被静默吞掉
- **修复**：catch → `console.error`；新增 `[QR poll]` 日志记录每次轮询的响应码；添加 default 分支处理未知状态码
- **涉及**：`login-service.ts`、`LoginPanel.tsx`

#### Bug 5 🟡 页面文字不可选择/复制
- **表现**：浏览器中无法选中文字、无法复制
- **根因**：`index.css` 全局 `body { user-select: none }`（为 Electron 标题栏拖拽设置的），同时影响了浏览器版
- **修复**：删除全局 `user-select: none`；TitleBar 已有 `select-none` (Tailwind) 不受影响
- **涉及**：`index.css`

#### Bug 6 🟡 封面一直卡在"加载中…"
- **表现**：spinner 动画永远不结束
- **根因**：`<img style="display:none">` 在大多数浏览器中不会触发网络请求，`onLoad` 永远不触发
- **修复**：`<img>` 始终 `display: block`，loading/error 状态用绝对定位覆盖层叠加
- **涉及**：`VideoCard.tsx`

#### Bug 7 🟡 暗色主题：多处对比度问题
- **表现**：用户反馈暗色主题下文字看不清
- **审计方法**：并行审查了 11 个组件文件，逐一排查硬编码色值 / 未定义 CSS 变量
- **修复 11 处**（详见第四节问题 3）
- **涉及**：`EpisodeList.tsx`、`HistoryPage.tsx`、`SettingsPage.tsx`、`StatusBar.tsx`、`FfmpegBanner.tsx`、`InputBar.tsx`

#### Bug 8 🔴 QR 登录：HttpOnly Cookie 导致扫码成功但无法提取 Cookie
- **表现**：扫码后显示"登录成功"但页面没有变为"已登录"状态，刷新后仍提示未登录
- **根因**：Vite 代理的 `onProxyRes` 处理了 `Secure`/`SameSite` 但**没有剥离 `HttpOnly`**。B 站 `SESSDATA` cookie 带有 `HttpOnly` → 浏览器 `document.cookie` 无法暴露给 JS → `extractBiliCookies()` 返回空字符串

- **涉及**：`vite.config.ts`、`LoginPanel.tsx`

#### Bug 9 🟡 解析后标题不显示（防御性加固）

- **修复**：`apiGet` 后增加显式 null 检查 `if (!detail || typeof detail !== 'object') throw`，确保正确触发 WBI fallback
- **涉及**：`bilibili-api.ts`

#### Bug 10 🟡 下载速度：4 个性能问题

## 四、关键问题深度分析

### 问题 1：zustand v5 无限重渲染 — `Object.is` 陷阱

**背景**：zustand v5 内部用 `React.useSyncExternalStore(subscribe, getSnapshot)` 做状态订阅。每次 `set()` 调用遍历所有 listener → React 调 `getSnapshot()` → `Object.is(prev, next)` 判断是否重渲染。

**触发条件**：
```tsx
// ❌ 每次渲染创建新对象 → Object.is 永远 false → 死循环
const { addItem, pauseItem } = useDownloadStore((s) => ({
  addItem: s.addItem,
  pauseItem: s.pauseItem,
}))
```

箭头函数体 `({...})` 每次调用返回**全新对象引用**。即使 `s.addItem` 和 `s.pauseItem` 是同一个函数引用（store 不变），包含它们的对象 `{addItem, pauseItem}` 是新引用 → `Object.is` 判定"变了" → re-render → 又创建新对象 → 无限循环。

这和 persist 中间件无关。persist 只是**触发者**（它调了一次 `set()`），真正导致循环的是 selector 返回新对象。

**正确做法**：
```tsx
// ✅ 每个 selector 返回稳定值（函数引用/原语）
const addItem = useDownloadStore((s) => s.addItem)  // 函数引用始终相同
const pauseItem = useDownloadStore((s) => s.pauseItem)

// ✅ 或需要对象时用 useShallow（浅比较替代 Object.is）
const { a, b } = useStore(useShallow(s => ({ a: s.a, b: s.b })))
```

### 问题 2：HttpOnly Cookie — 代理与浏览器安全策略的冲突

**完整的"扫码成功但无法登录"链路**：

```
1. 用户扫码 → B 站 passport API 返回 code:0 + Set-Cookie: SESSDATA=xxx; HttpOnly; Secure; Domain=.bilibili.com
2. Vite 代理的 onProxyRes 处理 Set-Cookie：
   ├── cookieDomainRewrite: .bilibili.com → localhost ✅
   ├── .replace(/Secure/gi, '') ✅
   ├── .replace(/SameSite=.../gi, '; SameSite=Lax') ✅
   └── .replace(/HttpOnly/gi, '') ❌ 缺失！
3. 浏览器收到 Set-Cookie → 设置 cookie 在 localhost 域，但标记为 HttpOnly
4. pollQrStatus() 返回 success → extractBiliCookies() 读 document.cookie
5. document.cookie 对 HttpOnly cookie 返回空字符串 ❌
6. cookieStr = '' → 登录流程静默失败
```

**修复**：代理剥离 HttpOnly（允许 JS 读取）+ LoginPanel 增加 fallback（兜底）。

**安全考量**：B 站使用 HttpOnly 是为防止 XSS 窃取 cookie。在我们的场景中，Vite 代理仅在**本地开发环境**运行，不存在 XSS 攻击面。Electron 生产模式下由主进程管理 cookie，不经过浏览器 `document.cookie`。

### 问题 3：暗色主题系统性审计 — 方法

**审计维度**：
1. **CSS 变量存在性** — 引用是否定义在 `tokens.css`？未定义则回退到 `inherit`/浏览器默认
2. **硬编码色值** — `#xxx`、`rgb(...)` 不管主题如何变化始终不变
3. **对比度** — 前景色 vs 背景色 ≥ 4.5:1（WCAG AA 普通文本）/ 3:1（大文本）
4. **Tailwind 类** — `text-gray-xxx`、`bg-gray-xxx` 硬编码不响应主题切换

**审计结果**：2 个 CRITICAL（未定义变量）、2 个 HIGH（对比度不达标）、2 个 MEDIUM（不随主题适配的硬编码）、5+ 个 LOW（hover 效果主题偏向）。

### 问题 4：下载队列状态机 Bug

**resumeItem 为什么不工作**：

```
processQueue 查找: items.filter(d => d.status === 'queued')     ← 只认 'queued'
resumeItem 设置:  { ...d, status: 'downloading' }                ← 设的是 'downloading'

result: 恢复的下载永远不会被 processQueue 拾取 → 永远卡住
```

**对比 retryItem（正确实现）**：
```
retryItem 设置:  { ...d, status: 'queued' }  ← 回到队列排队
processQueue 找到 → processDownload 启动 → 实际下载
```

这是一个典型的状态机设计错误：`resumeItem` 试图越过调度器直接设置执行状态，但调度器（`processQueue`）不理解这个状态跳变。
