## 二、逐问题详解

### 2.1 配色丢失 — warning-bg token + 下载弹窗图标

#### 用户反馈

> "我怎么感觉你把我原来一些配色换掉了或者没有显示出来"
> "点击加入下载队伍跳出来的窗口你的配色"

#### 问题 1：`--color-warning-bg` 是灰色

**根因**：`src/styles/tokens.css` 在 v7.2.0 时期就把 `--color-warning-bg` 设成了中性灰。但之前被 DownloadChoiceDialog 的 CSS fallback 遮盖了（`var(--color-warning-bg, #fef3c7)`），R19 去掉 fallback 后暴露。

对比同一文件里其他语义 token：
```css
/* ✅ 正确 — 绿色/红色背景 */
--color-success-bg: var(--success-50);       /* 浅绿 */
--color-error-bg:   var(--error-50);         /* 浅红 */

/* ❌ 错误 — 中性灰（修复前） */
--color-warning-bg: rgba(128, 128, 128, 0.06);  /* 几乎不可见 */
```

**影响范围**（6 个组件）：
- `DownloadChoiceDialog.tsx` (2 处：合并警告图标背景 + 底部提示横幅)
- `ConfirmDialog.tsx`（警告弹窗图标背景）
- `FfmpegBanner.tsx`（FFmpeg 状态提示）
- `Toast.tsx`（warning 级别通知）
- `AboutPage.tsx`（关于页提示卡片）

**修复**：3 处 token 值改为琥珀色
```css
/* 亮色模式 */
--color-warning-bg: var(--warning-50);         /* #FFFBEB 琥珀白 */

/* 暗色模式（data-theme + @media） */
--color-warning-bg: rgba(245, 158, 11, 0.12); /* 琥珀色半透明 */
```

#### 问题 2：下载弹窗选项图标是灰色

`DownloadChoiceDialog` 的 `ChoiceButton` 组件中，4 个选项按钮的图标用了 `var(--text-secondary)`（灰色），和单流模式「下载视频」按钮的蓝色图标不一致。

```tsx
// 修复前
<span style={{ color: 'var(--text-secondary)' }}>{icon}</span>

// 修复后
<span style={{ color: 'var(--color-accent)' }}>{icon}</span>
```

### 2.2 浏览器模式 CDN 下载 Access Denied — 核心修复

#### 用户反馈

> "一点击下载就是 404"
> 实际错误信息：`Access Denied — You don't have permission to access "http://upos-hz-mirrorakam.akamaized.net/..." on this server`

#### 根因分析

Bilibili 的 CDN（Akamai）对下载请求做了 Referer + Origin 校验，必须来自 `https://www.bilibili.com`。

**下载链路**：
```
用户点击下载 → resolveDownloadUrl() 获取 CDN URL ✓（走 API 代理，成功）
              → downloadWithProgress() fetch CDN URL ✗（浏览器禁止设 Origin）
```

关键问题在 `downloadWithProgress`：
```ts
const res = await fetch(url, {
  headers: {
    Origin: 'https://www.bilibili.com',   // ← 浏览器禁止！fetch 不允许手动设 Origin
    Referer: referer,
  },
})
```

浏览器安全策略：`Origin` 是**禁止头**（forbidden header），`fetch` 无论如何设置都会被浏览器静默忽略，实际发的是 `Origin: http://localhost:5173`。CDN 校验不通过 → Access Denied。

**注意**：代码里还有一个 `<a>` 标签降级方案（`triggerBrowserDownloadDirect`），但 `<a>` 标签导航请求同样带 `Referer: localhost:5173`，CDN 一样会拒绝。所以浏览器模式完全无法下载。

#### 解法：Vite 下载代理

核心思路：**让下载请求走本地服务器转发**，服务端可以设任意请求头。

**步骤 1**：`vite.config.ts` 新增 `downloadProxyPlugin`，挂载 `/api/proxy-download?url=<encoded_cdn_url>` 端点：

```typescript
function downloadProxyPlugin() {
  return {
    name: 'download-proxy',
    configureServer(server) {
      server.middlewares.use('/api/proxy-download', async (req, res) => {
        const targetUrl = new URL(req.url, 'http://localhost').searchParams.get('url')
        // 以服务端身份 fetch CDN（可设任意头）
        const response = await fetch(targetUrl, {
          headers: {
            Referer: 'https://www.bilibili.com',
            Origin: 'https://www.bilibili.com',
            'User-Agent': 'Mozilla/5.0 ...',
          },
        })
        // 流式转发响应体（保留进度追踪能力）
        const reader = response.body.getReader()
        while (true) {
          const { done, value } = await reader.read()
          if (done) { res.end(); break }
          res.write(value)
        }
      })
    },
  }
}
```

**步骤 2**：`download-manager.ts` 的 `downloadWithProgress` 函数根据环境切换 URL：

```typescript
// 浏览器模式 → 走本地代理（同源请求，绕过 CORS + CDN 防盗链）
// Electron 模式 → 直连（可设任意头，不受影响）
const fetchUrl = !isElectron()
  ? `/api/proxy-download?url=${encodeURIComponent(url)}`
  : url
```

**效果**：
- ✅ 浏览器模式：请求走 `localhost:5173/api/proxy-download`（同源，无 CORS 问题）→ 服务端转发到 CDN → CDN 看到 `Origin: bilibili.com` → 放行
- ✅ 进度追踪保留：服务端流式转发，`ReadableStream` 仍然可以分段读取 + 报告进度
- ✅ Electron 模式：不受影响，继续直连

#### 这个模式的局限（告知接手者）

1. **仅 dev 模式可用**：代理依赖 Vite dev server。生产浏览器部署（Vercel/GitHub Pages）无法使用，需要单独的后端代理服务。
2. **大文件内存占用**：代理将整个文件加载到 Node.js 内存再转发。超大文件（>2GB）可能有问题，但 B站视频分片通常 <500MB。
3. **单用户场景**：代理没有做并发限制，多用户同时下载会竞争带宽。

## 八、后续接手建议

### 如果要做浏览器线上部署（任务 #3）

当前 CDN 下载代理是 Vite 插件，**仅开发模式可用**。线上部署需要：
1. 在 Vercel/Cloudflare Workers 上写一个独立的代理端点（Serverless Function）
2. 或部署一个简单的 Node.js 代理服务
3. 修改 `downloadWithProgress` 的 `fetchUrl` 逻辑，让它能自动检测环境并切换到线上代理

### 如果要继续打磨用户体验

- **Electron 单流下载也加代理？** — 目前 Electron 直连 CDN 正常，但国产杀软/防火墙可能拦截。如果遇到问题，可以考虑让 Electron 也走类似代理。
- **下载速度优化** — 代理模式比直连多一跳（本地转发），速度有轻微损耗。可以考虑加并发分片下载。
