### 3.2 下载选择弹窗 — 全格式覆盖

**问题**：原弹窗 `showDownloadChoiceDialog()` 只在 `isDash && hasVideo && hasAudio` 时弹出。BV1GJ411x7h7 等 FLV 单流格式视频直接跳过弹窗，静默按 `video-only` 下载。用户完全看不到任何选择。

**根本原因**：`processDownloadBrowser:537` 的条件判断：
```ts
if (isDash && hasVideo && hasAudio) {
    mode = await showDownloadChoiceDialog(safeName)
}
```
FLV 视频 `urlResult.format === 'flv'`，`isDash` 为 false，弹窗被跳过。

**修复**（`download-manager.ts`）：

1. **弹窗触发条件放宽**：浏览器模式下，`processDownloadBrowser` 始终调用 `showDownloadChoiceDialog`，不再判断格式
2. **弹窗内容自适应**：重写 `showDownloadChoiceDialog`，接受 `isDash/hasVideo/hasAudio` 参数：

| 视频类型 | 弹窗内容 |
|---------|---------|
| DASH + 双流（video+audio） | 三选一：📹仅视频 / 🎵仅音频 / 📦分别下载 |
| FLV 单流（只有 video） | 确认页：📹下载视频（提示"单流格式，音视频已合并，无需额外操作"） |
| DASH + 仅视频（无独立音轨） | 确认页：📹下载视频（提示"DASH 单视频流，无独立音轨"） |
| 纯音频（无 video） | 直接返回 `audio-only`，不弹窗 |

3. **移除硬编码 fallback 颜色**：弹窗 DOM 中原有大量 `#1e1e2e`/`#252536`/`#888` 等 fallback，全部改为纯 CSS 变量引用

### 3.4 下载队列持久化

**问题**：`downloadStore` 纯内存存储，刷新浏览器后整个下载队列消失。

**修复**（`downloadStore.ts`）：
- 包裹 `persist` 中间件（与 `userPrefsStore` 相同模式）
- 存储名 `bibilidown-downloads`
- `skipHydration: true` — 手动在 `App.tsx` 中调用 `rehydrate()`

**接线**（`App.tsx:115`）：与 `userPrefsStore`/`historyStore` 一起调用 `useDownloadStore.persist.rehydrate()`

### 3.5 下载重试逻辑

**问题**：`maxRetries` 设置项存储但从未使用，下载失败直接标记 failed。

**修复**（`download-manager.ts:253-270`）：
- `processDownload` 外层包裹 `for (let attempt = 0; attempt < maxRetries; attempt++)` 循环
- 失败时更新 speed 为 `重试中 (1/3)...`
- 重试间隔 2 秒
- 智能分类错误：
  - 重试：网络错误（`TypeError`/`fetch failed`/`NetworkError`/`Failed to fetch`）
  - 不重试：用户取消（`AbortError`）、权限错误（401/403）、非网络错误（FFmpeg 合并失败等）

### 3.11 弹幕 & 字幕 API

**修复**（`bilibili-api.ts`）：

`fetchDanmaku(cid: number): Promise<string>`：
- GET `https://api.bilibili.com/x/v1/dm/list.so?oid={cid}`
- 返回 XML 格式弹幕数据
- 失败返回空字符串（弹幕是可选功能，不阻塞下载）

`fetchSubtitle(bvid: string, cid: number): Promise<string>`：
- 第一步：GET `/x/player/v2?bvid=&cid=` → 取 `data.subtitle.subtitles` 数组
- 第二步：选第一个字幕 URL（通常为中文），`//` 补全为 `https://`
- 第三步：下载字幕 JSON → 转为 SRT 格式（`body[].from` → 开始时间，`body[].to` → 结束时间，`body[].content` → 文本）
- 失败返回空字符串

## 四、核心问题深度分析

### 4.1 下载选择面板不显示的根因

**问题链**：

```
用户点击下载
  ↓
processDownload → resolveDownloadUrl(bvid, cid, qn)
  ↓
B站 API 返回: format: 'flv' 或 format: 'dash' 但无 audio 流
  ↓
processDownloadBrowser: isDash = urlResult.format === 'dash' && hasAudio  → false
  ↓
if (isDash && hasVideo && hasAudio) → SKIP 弹窗
  ↓
mode 静默默认为 'video-only'
  ↓
用户看到：下载开始 → 下载完成，完全没有选择环节 ❌
```

**为什么 BV1GJ411x7h7 等视频是 FLV**：
- B站部分老视频仅提供 FLV 格式（单一文件，音视频已合并）
- 即使 API 请求 DASH 格式（`fnval=4048`），B站仍可能降级返回 FLV
- 登录状态不影响格式类型，旧视频就是旧视频

**解决方案**：
1. 浏览器模式下**无论什么格式都弹窗**，让用户知道发生了什么
2. DASH 双流 → 三选一
3. FLV/DASH 单流 → 确认页（说明单流/无需选择）
4. 纯音频 → 自动处理

---

### 4.2 设置页"假开关"问题的系统性解决

**问题根源**：代码迭代中功能先于设置页开发。设置页 UI 由早期阶段搭好框架，但对应的下载引擎逻辑是后来逐步添加的，从未回过头来接线。

**本轮做了三层处理**：
1. **能接的接上**：`preferredQuality` / `filenameTemplate` / `maxRetries` / `downloadDanmaku` / `downloadSubtitle`（5 个）
2. **不能接的标出来**：`downloadNotify` / `openFolderAfterDownload` / `downloadPath` → "仅桌面版" badge（3 个）
3. **待考虑的去留**：`preferredFormat` — 用户实际不需要选格式，始终用 DASH 最佳（1 个）

---

### 4.3 浏览器 vs 桌面版 — 系统性差异

这是用户反复强调的核心关切。本轮做了系统性的"双端可见化"：

| 差异维度 | 浏览器 | 桌面版 (Electron) |
|---------|--------|-------------------|
| 音视频合并 | ❌ 需手动选择 | ✅ FFmpeg 自动合并 |
| 文件保存 | `<a download>` 默认下载文件夹 | 原生对话框 + 任意路径 |
| 系统通知 | ❌ 不可用 | ✅ Notification API |
| 打开文件夹 | ❌ 不可用 | ✅ shell.openPath |
| 窗口控制 | 隐藏 | 最小化/最大化/关闭 |
| 标题栏 | BibiliDown + 图标 | 完整拖拽区 + 窗口控制 |
| 下载队列持久化 | localStorage | localStorage |
| 下载进度 | 跳变 0%→100% | 实时进度条 |
| 状态栏 | 🌐 浏览器模式 | 🖥️ 桌面版 |
| 托盘最小化 | ❌ 隐藏设置项 | ✅ 可用 |
| 剪贴板检测 | ✅ 可用 | ✅ 可用 |

## 七、核心经验 & 注意事项

### 7.1 下载弹窗的教训：无声失败是最差的 UX

原弹窗设计中，FLV 视频静默跳过弹窗、默认用 `video-only` 下载。代码逻辑完全正确，但用户不知道发生了什么 —— "为什么我没看到选项？是 Bug 吗？"

**教训**：任何不弹窗的"默认行为"都应该有用户可见的确认。宁可多弹一个确认页（"FLV 单流格式，无需选择"），也不要让用户困惑。

### 7.2 设置页接线要跟随功能开发

设置页和功能逻辑是两拨不同时间开发的代码。功能先铺好（所有设置项默认值在 store 中），然后下载引擎逐步完善。但引擎完善后没人回头把设置项接上。本轮专项做了 5 个接线，仍有 4 个待接。

**教训**：设置页不要提前搭完整框架。每个设置项应该在对应功能实现的同时接线，避免"假开关"积压。

### 7.3 Zustand v5 persist 的 skipHydration 模式

本项目使用 `skipHydration: true` + 手动 `rehydrate()` 模式（在 App.tsx useEffect 中集中调用所有 store 的 rehydrate）。这是为了避免 zustand v5 的自动 hydration 在某些场景下触发无限重渲染。

**所有新的 persist store 必须遵循此模式**：
```ts
persist(storeCreator, {
  name: 'storage-key',
  skipHydration: true,
  partialize: (state) => ({ /* 只序列化需要的字段 */ }),
})
```

然后在 `App.tsx:115` 附近添加 `.persist.rehydrate()`。

### 7.4 浏览器模式的双重限制

浏览器有两个硬性限制无法绕过：
1. **文件系统**：`<a download>` 只能保存到浏览器默认下载文件夹，无法指定路径。File System Access API 可以写入任意目录，但需要用户授权且兼容性有限。
2. **系统通知**：Web Notification API 需要用户授权，且体验远不如 Electron 原生通知。

这些不需要"修复"，只需要**明确告知用户**。本轮通过 StatusBar 模式指示器和设置页 badge 做了这一步。

### 7.5 本轮使用的并行 Agent 模式

本轮同时启动了 5 个 Claude Code Agent 并行修改代码：
1. SettingsPage 接线（badge + preferredQuality + filenameTemplate）
2. 下载持久化 + 重试
3. StatusBar + InputBar
4. 弹幕 + 字幕 API
5. 主题色修复

每个 Agent 操作**互不重叠的文件集合**，避免编辑冲突。所有 Agent 完成后统一用 `npx tsc --noEmit` 验证编译。

**关键**：必须提前规划好每个 Agent 的文件范围，确保零交集。如果两个 Agent 都要改同一个文件，先后顺序执行。
